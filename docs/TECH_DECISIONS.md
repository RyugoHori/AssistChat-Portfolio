# 技術選定の理由

このドキュメントでは、AssistChatで採用した各技術の選定理由を説明します。
単なる「流行っているから」ではなく、**現場の制約と要件**に基づいた判断です。

---

## 最重要：ローカル完結アーキテクチャの理由

### 背景

このプロジェクトは、**会社の正式なプロジェクトではなく、個人の現場改善活動**としてスタートしました。

| 状況 | 制約 |
|------|------|
| 会社方針でないプロジェクト | 機密データを外部サービスに送信できない |
| 製造業の保全記録 | 設備情報、故障パターンは機密性が高い |
| 個人の裁量で進行 | 会社承認なしに外部サービス契約は不可 |

### 結果：全ての技術をローカル完結に

| 機能 | 選定技術 | 外部API依存 |
|------|----------|-------------|
| Embedding | sentence-transformers | ❌ なし |
| ベクトル検索 | FAISS | ❌ なし |
| キーワード検索 | rank-bm25 | ❌ なし |
| Re-ranker | Cross-Encoder | ❌ なし |
| 形態素解析 | MeCab + unidic-lite | ❌ なし |
| LLM回答生成 | 未実装 | - |

**外部サービス依存は一切ない。データが社外に出ることはありません。**

> 後に会社方針でDatabricksが正式採用されましたが、それは「会社が承認したクラウド基盤」だからこそ可能になったことです。個人開発時にPinecone等を使っていたら、機密情報の取り扱いとして問題になっていた可能性があります。情報システム部の方々にローカル開発の技術選定について認可を得たのちに個人開発開始しました。

## 検索エンジン

### なぜハイブリッド検索か？

| 検索方式 | 長所 | 短所 |
|----------|------|------|
| ベクトル検索のみ | 意味的な類似性を捉える | キーワード完全一致が弱い |
| キーワード検索のみ | 完全一致に強い | 言い換えに対応できない |
| **ハイブリッド** | 両方の長所を活かせる | 実装が複雑 |

製造業の保全記録では、以下の両方が重要です：
- **意味検索**: 「モーターが動かない」→「駆動系の停止」
- **キーワード検索**: エラーコード「E-1234」の完全一致

### FAISS vs 他のベクトルDB

| 選択肢 | 評価 | 理由 |
|--------|------|------|
| **FAISS** | ✅ 採用 | 軽量、高速、**ローカル実行可能** |
| Pinecone | ❌ 不採用 | クラウド依存、機密データを外部に送れない |
| Weaviate | ❌ 不採用 | クラウド版は同上、セルフホストは運用コスト |
| Milvus | ❌ 不採用 | オーバースペック、運用が重い |
| Chroma | △ 検討 | 軽量だがFAISSより遅い |

**決定理由**: 
- 機密データを外部に出せない制約 → ローカル完結必須
- 1万件程度のデータ量ならFAISSで十分（10万件まで対応可能）
- Metaが開発、実績豊富で信頼性が高い

### BM25 vs Elasticsearch

| 選択肢 | 評価 | 理由 |
|--------|------|------|
| **rank-bm25** | ✅ 採用 | 軽量、Pythonネイティブ、インメモリ動作 |
| Elasticsearch | ❌ 不採用 | 重い、運用コスト、500件程度には過剰 |
| Whoosh | △ 検討 | 日本語対応が弱い |

**決定理由**: 
- 最初はElasticsearchを検討したが、500件程度ならインメモリで十分と判断
- Dockerイメージを軽量に保てる
- 依存関係がシンプル

## 埋め込みモデル

### sentence-transformers を選んだ理由

| 選択肢 | 評価 | 理由 |
|--------|------|------|
| **paraphrase-multilingual-mpnet-base-v2** | ✅ 採用 | 日本語対応、軽量、**ローカル実行可能** |
| OpenAI Embeddings | ❌ 不採用 | API依存、機密データを外部に送れない |
| Cohere Embed | ❌ 不採用 | 同上 |
| E5-large | △ 検討 | 精度は高いがサイズ大（2GB超） |
| multilingual-e5-small | △ 検討 | 将来的に検討の余地あり |

**決定理由**: 
- 日本語を含む50言語に対応
- 768次元で十分な表現力
- モデルサイズ約1GB、メモリ効率が良い
- **ローカル実行可能**（最重要）

## Re-ranker

### Cross-Encoder を選んだ理由

| 選択肢 | 評価 | 理由 |
|--------|------|------|
| **mmarco-mMiniLMv2** | ✅ 採用 | 多言語対応、軽量、**ローカル実行可能** |
| ms-marco-MiniLM | △ 検討 | 英語のみ |
| Cohere Rerank | ❌ 不採用 | API依存、機密データを外部に送れない |

**決定理由**:
- 日本語対応の多言語モデル
- 推論速度と精度のバランスが良い
- **ローカル実行可能**

### なぜRe-rankingが必要か？

```
Bi-Encoder (FAISS/BM25):
  Query → [Encoder] → Vector
  Doc   → [Encoder] → Vector
  → 独立にエンコード、高速だが精度に限界

Cross-Encoder (Re-ranker):
  (Query, Doc) → [Encoder] → Score
  → ペアでエンコード、低速だが高精度
```

**戦略**: Bi-Encoderで候補を絞り（Top-10）、Cross-Encoderで精密に順位付け

実測で**MRRが29%向上**しました。計算コストは増えるが、精度向上の効果が大きい。

## 日本語処理

### MeCab + unidic-lite を選んだ理由

| 選択肢 | 評価 | 理由 |
|--------|------|------|
| **MeCab + unidic-lite** | ✅ 採用 | 高精度、辞書が軽量 |
| Janome | △ 検討 | 辞書が古い、製造業用語の分割精度が低い |
| SudachiPy | △ 検討 | 重い、Dockerイメージが肥大化 |
| spaCy (ja) | △ 検討 | MeCabベースなので同等 |

**決定理由**:
- 製造業用語（「チャッキングミス」「リミットスイッチ」等）の分割精度が高い
- unidic-liteは約50MBで軽量
- Dockerコンテナ内で安定動作

## フロントエンド

### Next.js 14 (App Router) を選んだ理由

| 選択肢 | 評価 | 理由 |
|--------|------|------|
| **Next.js 14** | ✅ 採用 | 採用実績多、優れたDevExでサポート安定、next.jsの経験値をさらに高めたい |
| Remix | △ 検討 | 大規模安定向けだが採用実績が少ない オーバースペック |
| Vite + React | △ 検討 | 構成自由度高い反面SSR等自前設定が煩雑化する可能性あり |
| Vue/Nuxt | △ 検討 | 小規模メインで採用実績が少ない、TSサポートやエコシステム未熟 |

**決定理由**:
- React Server Componentsによるパフォーマンス
- App Routerの直感的なルーティング
- 転職市場での需要が高い

### shadcn/ui を選んだ理由

| 選択肢 | 評価 | 理由 |
|--------|------|------|
| **shadcn/ui** | ✅ 採用 | カスタマイズ性、保守性　見た目が好き |
| MUI | △ 検討 | 重い、バンドルサイズが大きい |
| Chakra UI | △ 検討 | 良いがshadcnの方がトレンド |
| Headless UI | △ 検討 | スタイリングが手間 |

**決定理由**:
- コンポーネントをコピーして所有できる（ロックインなし）
- Radix UIベースでアクセシビリティ対応
- Tailwind CSSとの相性が良い
- 最初はMUI使おうとしたけど、バンドルサイズが大きすぎてやめた

## バックエンド

### FastAPI を選んだ理由

| 選択肢 | 評価 | 理由 |
|--------|------|------|
| **FastAPI** | ✅ 採用 | 高速、型安全、自動ドキュメント |
| Flask | △ 検討 | 型ヒントサポートが弱い |
| Django | ❌ 不採用 | オーバースペック、ORMは不要 |
| Express.js | △ 検討 | Pythonエコシステム（ML/NLP）と分離してしまう |

**決定理由**:
- Pydanticによる型安全なリクエスト/レスポンス
- OpenAPI (Swagger) の自動生成
- async/await のネイティブサポート
- Python MLエコシステムとの親和性

## インフラ

### Docker Compose を選んだ理由

| 選択肢 | 評価 | 理由 |
|--------|------|------|
| **Docker Compose** | ✅ 採用 | シンプル、ローカル開発に最適 |
| Kubernetes | ❌ 不採用 | 深く学びたかったがオーバースペック |
| ローカル直接実行 | ❌ 不採用 | MeCab等の様々環境差異が発生リスク有 |

**決定理由**:
- 開発環境の統一
- `docker-compose up` で一発起動
- 本番デプロイへの移行も容易

## 採用しなかった技術

### LangChain

**不採用理由**:
- 抽象化レイヤーが厚く、デバッグが困難
- 以前ITスタートアップで使っていたが、自分自身ロジックを深く理解できていなかった
- 今回は「RAGの仕組みを根本から理解する」ことも目的だったので、自前実装を選択

### クラウドベクトルDB (Pinecone, Weaviate)

**不採用理由**:
- **最重要**: 個人開発時は機密データを外部に出せなかった
- 会社方針でないプロジェクトで外部サービス契約はできない
- FAISSで1万件程度は余裕で対応可能

### PostgreSQL / データベース

**不採用理由**:
- 1万件程度の読み取り専用データなら、ファイルベースで十分
- トランザクション管理は不要
- 本番化時にPostgreSQL導入を検討していたが、会社方針でDatabricks統一となり未実装

### GraphQL

**不採用理由**:
- REST APIで十分な要件
- 実装の複雑さに見合うメリットがない
- クライアント（Next.js）との通信はシンプルなJSON-RPCで十分

## データ規模の考慮

### 現状（ポートフォリオ版）

- 500件のダミーデータ
- 会社の情報は匿名化済み

### 本番想定

- 全ライン × 過去8年分 = 約10,000件
- FAISSは10万件程度まで問題なく対応可能
- 30人同時検索でもボトルネックにならない

**500件はポートフォリオ用であり、現場での仮運用は10,000件規模を想定して実装。**

## まとめ：技術選定の軸

1. **機密保護**: 製造業の保全記録は機密性が高い → ローカル完結必須
2. **状況適応**: 個人開発時と会社承認後で最適解は異なる
3. **実用性**: オーバースペックを避け、要件に見合った技術を選択
4. **学習効果**: 抽象化に頼らず、仕組みを理解できる構成

> **要件定義/設計から技術選定まで視野を広く持つことの重要性を体感した。**
