## このプロジェクトを作った理由

### 現場で感じていた課題

工場で設備保全をやっていると、夜間に故障が発生することがあります。
私は昼勤なので、夜間の故障は3交代勤務の保全班が対応します。

**ここに問題がありました：**

| 対象 | 課題 |
|------|------|
| 3交代班 | 全課の全ラインを横断的に担当するため、特定設備の知見が浅い |
| 若手保全マン | 初めて経験する故障パターンに対応できない |
| ベテラン | 課を異動すると、その課の過去の故障履歴や癖を知らない |

その結果、原因究明に時間がかかり設備停止が120分を超えると「重大故障」判定になります。
担当ラインで重大故障になってしまうとなぜなぜ分析や再発防止対策などの対応が発生してしまい業務量が膨れあがってしまいます。


**過去に何度もあったケース：**
> ベテラン不在時若手のみで突発故障対応 → 若手の知見では原因究明に時間を要す → 気が付いたら設備停止時間が長くなってしまい生産性低下。→ベテランならすぐ原因分かってたのに...

> 夜間に故障発生 → 3交代班が原因わからず苦戦 → 120分超えて重大故障 → 翌日調べてみたら「これ前にも同じ現象起きてるじゃ～ん。過去記録の通り○○を○○したら5分で直せたのに...」

過去の故障記録を素早く検索できれば、こういったケースを防げる。
尚且つ故障対応表を毎度愚直に書いているにもかかわらず、故障対応中に過去対応表を見る文化が無く上手く活用されていないと感じた。
これがこのアプリを作ろうと思ったきっかけです。

### 個人的なモチベーション

正直なところ、現場課題の解決だけじゃなく、こんな気持ちもありました：

- ITスキルを磨いて、IT人材としての地位を築きたい
- ITスタートアップではRAG開発案件にも携わっており、その際担当工程は違うもののLangChainの抽象化が多く、自分自身ロジックを深く理解できていなかった
- 個人開発でじっくり仕組みを根本から理解してみたいという純粋な好奇心

## 開発の経緯

### フェーズ1: プロトタイプ開発（2025年2月〜3月）

最初はシンプルなPythonスクリプトとStreamlitでプロトタイプを作成しました。

```
初期構成:
├── ingest.py          # CSVデータの読み込み（実データ特定ラインサンプル300件程度）
├── search_engine.py   # FAISS検索（意味検索）
├── app.py             # Streamlit UI（超簡易ＵＩ）
└── requirements.txt
```

#### 直面した課題

1. **専門用語の検索精度が低い**
   - 「チャッキングミス」「リミットスイッチ」などの製造業特有の用語
   - 汎用的な埋め込みモデルでは意味を捉えきれない

2. **エラーコードの完全一致が必要**
   - 「E-1234」のようなエラーコードは、ベクトル検索では類似コードと混同される
   - キーワード検索との併用が必要不可欠だとわかった

#### 学び

> ベクトル検索だけでは不十分。キーワード検索との組み合わせが必要。

---

### フェーズ2: ハイブリッド検索の導入（2025年3月〜4月）

#### 技術選定  
詳細 → [技術選定](docs/TECH_DECISIONS.md)

| 要件 | 選定技術 | 理由 |
|------|----------|------|
| ベクトル検索 | FAISS | 高速、メモリ効率、実績豊富 |
| キーワード検索 | BM25 (rank-bm25) | 軽量、日本語対応容易 |
| 日本語形態素解析 | MeCab + unidic-lite | 製造業用語の分割精度 |
| 埋め込みモデル | sentence-transformers (multilingual) | 日本語対応、軽量 |

#### Reciprocal Rank Fusion (RRF)

2つの検索結果を統合するために、RRFアルゴリズムを採用しました。

```python
# RRFスコア計算
Score = Σ(1 / (k + rank))  # k=60が最適（論文より）
```

**なぜRRFか？**
- スコアの正規化が不要（ランクベース）
- 実装がシンプル
- 論文で有効性が実証されている

#### 精度向上の取り組み

1. **MeCabの辞書カスタマイズ**
   - 製造業用語を含む辞書（unidic-lite）を採用
   - 将来的にはユーザー辞書の追加も検討

2. **Re-rankingの導入**
   - Cross-Encoderによる再順位付け
   - 計算コストは増えるが、精度が大幅に向上

#### 精度評価

各検索手法の効果を定量的に評価しました。

| 手法 | MRR@5 | Recall@5 | 改善率 |
|------|-------|----------|--------|
| Dense (FAISS) のみ | 0.48 | 0.62 | baseline |
| Sparse (BM25) のみ | 0.45 | 0.58 | -6% |
| Hybrid (RRF) | 0.67 | 0.78 | **+40%** |
| Hybrid + Re-ranker | **0.81** | **0.89** | **+69%** |
(仮運用時実データ1万件での実績)
> 評価スクリプトは `tools/evaluation/` に格納しています。

### フェーズ3: Webアプリ化とモノレポ移行（2025年4月〜6月）

#### なぜWebアプリ化したか

- **複数ユーザーでの利用**: Streamlitでは同時アクセスに制限がある
- **UIの柔軟性**: より複雑なフィルタリングUIが必要
- **将来の拡張性**: チャット機能やフィードバック機能や生産管理システム連携等を見据えて

#### 技術スタック選定

| レイヤー | 技術 | 選定理由 |
|----------|------|----------|
| Frontend | Next.js 14 (App Router) | 採用実績数、優れたDevEx、next.jsの経験値をさらに高めるため |
| UI | shadcn/ui + Tailwind CSS | カスタマイズ性　保守性　見た目が好き |
| Backend | FastAPI | 高速、型安全、OpenAPI自動生成　個人開発でも最適 |
| インフラ | Docker Compose | 開発環境の統一、デプロイ容易 |

#### モノレポ構成の採用

```
AssistChat/
├── apps/
│   └── assist-chat/     # Next.js Frontend
├── packages/
│   └── rag-core/        # 共通RAGライブラリ
├── services/
│   └── rag-api/         # FastAPI Backend
└── tools/
    └── index-builder/   # インデックス構築ツール
```

**メリット:**
- `rag-core` をAPIサーバーとインデックス構築ツールで共有
- 依存関係の一元管理
- CI/CDパイプラインの簡素化

## 重要な技術的判断

### なぜLangChainを使わなかったか

以前ITスタートアップでRAG開発に携わった際、LangChainを使っていました。
便利だったけが、正直なところ**自分がRAGの仕組みを深く理解できていない**ことに気づきました。

| LangChainの問題点 | 自前実装のメリット |
|-------------------|-------------------|
| 抽象化が厚くてデバッグが困難 | 各ステップを自分でコントロールできる |
| 何かおかしいときに原因が追えない | 問題の切り分けが容易 |
| 学習には不向き | RAGの仕組みを根本から理解できる |

今回は「RAGを深く理解する」ことも目的だったので、あえて自前で実装しました。

### なぜクラウドベースのベクトルDB（Pinecone等）を使わなかったか

**これが一番重要なポイントです。**

開発当初、このプロジェクトは**会社の正式なプロジェクトではなく、個人の現場改善活動**でした。

| 制約 | 影響 |
|------|------|
| 会社方針でないのに機密データを外部に送れない | Pinecone、Weaviate等のクラウドサービスは使えない |
| 製造業の保全記録は機密性が高い | 外部サービス依存は避けるべき |
| 個人の裁量で進めていたプロジェクト | 会社承認なしに外部サービス契約はできない |

**だからこそ、全ての技術選定を「ローカル完結」にしました：**

| 機能 | 使用技術 | 外部API依存 |(情報システム部認可のもと)
|------|----------|-------------|
| Embedding | sentence-transformers | ❌ なし |
| ベクトル検索 | FAISS | ❌ なし |
| キーワード検索 | rank-bm25 | ❌ なし |
| Re-ranker | Cross-Encoder | ❌ なし |
| 形態素解析 | MeCab + unidic-lite | ❌ なし |

**外部サービス依存は一切ない。** データが社外に出ることはありません。

### なぜデータベース（PostgreSQL等）を使わなかったか

現状、データベースは使っていません。全てファイルベース（CSV → JSONL → FAISSインデックス）です。

**理由：**
- 1万件程度の読み取り専用データなら、FAISSインデックスをファイルで永続化する方式で十分
- トランザクション管理は不要
- 社内ネットワーク限定で利用者も限定されているため認証は未実装
- 仮運用中は口頭でのフィードバック収集で対応

本番化する場合は、PostgreSQLを導入してメタデータ管理とフィードバック保存に使う想定でしたが、後述の理由で未実装のままです。

## プロジェクトのその後

### 時系列

```
個人の現場改善として開発開始
    ↓
仮運用開始、各保全担当者に協力を頼み使用してもらう
    ↓
ICT班に認知される、技術的な情報提供を依頼される
    ↓
会社方針で正式にRAG開発が決定
  → 全国工場でDatabricks統一の方針
    ↓
このアプリは正式採用されず...（少し悲しい）
    ↓
しかし知見が評価され、兼務中のICT班のRAG開発担当に任命
    ↓
現在DatabricksでETLパイプライン等実装中、自作アプリの知見も存分に活用
```

### 技術選定の整合性

| フェーズ | 選定 | 理由 |
|----------|------|------|
| 個人開発時 | FAISS + ローカル完結 | 会社方針じゃないので機密データを外部に出せない |
| 会社方針後 | Databricks + Delta Lake | 会社が正式に承認したクラウド基盤 |

様々な記事/論文/AIをもとに**状況に応じた適切な技術選定ができた**と思っています。

## このプロジェクトで得たもの

### 技術的な成長

- RAGの仕組みを根本から理解できた
- ハイブリッド検索、Re-ranking、RRFなどの概念を実装レベルで習得
- ITスタートアップではCTO筆頭に他メンバーが事前に実装してくれていたため便利さが当たり前だと思っていたが
　技術選定然り、モノレポ構成、CI/CD、Dockerなどの開発プラクティスを自身で体験することでアーキテクチャの重　要性を再認識できた

### 社内での評価

- 愛知工場ではIT人材が極端に少なく現場レベルでのデジタル普及が遅れていたこともあり高評価
- ICT班に技術力を認められ、正式なRAG開発担当に任命
- 自分のアプリは採用されなかったが、知見は会社のプロジェクトに活かされている

### 学んだこと

> **「状況に応じた適切な技術選定」の重要性**
> 
> 個人開発時はローカル完結、会社承認後はクラウド基盤。
> ベストプラクティスに完全な正解はなく、状況に応じてそれぞれ慎重に技術選定することの大切さを学んだ
