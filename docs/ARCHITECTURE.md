# システムアーキテクチャ

## 概要

AssistChatは、RAG（Retrieval-Augmented Generation）パターンを採用した検索システムです。
ハイブリッド検索とRe-rankingにより、高精度な類似事例検索を実現しています。

### 設計思想

| 原則 | 実現方法 |
|------|----------|
| **ローカル完結** | 全ての処理をオンプレミスで実行、外部API依存なし |
| **機密保護** | データが社外に出ない構成 |
| **シンプルさ** | 過剰な抽象化を避け、デバッグしやすい構成 |
| **拡張性** | 将来のLLM統合やクラウド移行を見据えた設計 |

## コンポーネント詳細

### Frontend (apps/assist-chat)

| ディレクトリ | 役割 |
|-------------|------|
| `app/` | Next.js App Router ページ |

| ファイル | 役割 |
|---------|------|
| `main.py` | FastAPIアプリケーション、エンドポイント定義 |
| `api/models.py` | Pydanticモデル（リクエスト/レスポンス検証） |

| モジュール | 役割 |
|-----------|------|
| `search.py` | HybridSearcher（検索のオーケストレーション） |

| `sparse_index.py` | BM25インデックス管理 |
| `embeddings.py` | テキスト埋め込み生成 |
| `reranker.py` | Cross-Encoderによる再順位付け |
| `tokenization.py` | MeCabによる日本語トークナイズ |

## 検索フロー

```
1. ユーザーがクエリを入力
        │
        ▼
2. Frontend → Backend API (/api/search)
        │
        ▼
3. クエリの前処理
   - MeCabでトークナイズ
   - Sentence-Transformerで埋め込み生成
        │
        ├─────────────────┬─────────────────┐
        ▼                 ▼                 │
4. 　意味検索    　  キーワード検索   　 　　  │
    (FAISS)             (BM25)              │
    Top-K取得           Top-K取得            │
        │                 │                 │
        └────────┬────────┘                 │
                 ▼                          │
5. RRF (Reciprocal Rank Fusion)             │
   2つの結果をランクベースで統合           　  │
                 │                          │
                 ▼                          │
6. Post-filtering                           │
   メタデータによる絞り込み                   │
   (工場、ライン、設備、年度)                 │
                 │                          │
                 ▼                          │
7. Re-ranking (Cross-Encoder)               │
   クエリと各結果のペアをスコアリング          │
                 │                          │
                 ▼                          │
8. 最終結果を返却
```

## データフロー

### インデックス構築時

```
CSV (保全記録)
    │
    ▼ preprocess.py
Chunks (JSONLファイル)
    │
    ├─────────────────┬─────────────────┐
    ▼                 ▼                 │
FAISS Index       BM25 Index           │
(ベクトル)         (転置インデックス)     │
    │                 │                 │
    └────────┬────────┘                 │
             ▼                          │
    data/indices/ に保存                │
```

### 検索時

```
User Query
    │
    ▼
HybridSearcher.search()
    │
    ├─ dense_searcher.search()  → FAISS検索
    │
    ├─ sparse_searcher.search() → BM25検索
    │
    ├─ _reciprocal_rank_fusion() → 結果統合
    │
    ├─ _apply_filters() → フィルタリング
    │
    └─ reranker.rerank() → 再順位付け
    │
    ▼
Search Results (JSON)
```

## セキュリティ考慮事項

### ローカル完結アーキテクチャ

このシステムは**外部APIに一切依存しない**設計です。

| 機能 | 実装 | 外部通信 |
|------|------|----------|
| 埋め込み生成 | sentence-transformers（ローカル） | ❌ なし |
| ベクトル検索 | FAISS（ローカル） | ❌ なし |
| キーワード検索 | BM25（ローカル） | ❌ なし |
| Re-ranking | Cross-Encoder（ローカル） | ❌ なし |

**理由**: 製造業の保全記録は機密性が高く、開発当初は会社の正式プロジェクトではなく個人で開発していたため、外部サービスへのデータ送信は避ける必要がありました。

### その他のセキュリティ対策

1. **CORS設定**: 許可されたオリジンのみアクセス可能
2. **入力検証**: Pydanticによる厳密なリクエスト検証
3. **エラーハンドリング**: 内部エラーの詳細を外部に露出しない

---

## パフォーマンス最適化

1. **非同期処理**: FastAPIの非同期エンドポイント
2. **インデックスキャッシュ**: 起動時にメモリにロード
3. **Re-ranker候補数制限**: 上位10件のみ再順位付け

### 実測パフォーマンス

| 処理 | 所要時間 |
|------|----------|
| FAISS検索（500件） | ~20ms |
| BM25検索（500件） | ~30ms |
| Re-ranking（10件） | ~100ms |
| **合計** | **~150ms** |

## スケーラビリティ

### 現状の対応規模

| 項目 | 現状 | 上限目安 |
|------|------|----------|
| ドキュメント数 | 約1万件（実データの場合） | 10万件 |
| 同時接続数 | ~15人 | 30人程度 |
| インデックスサイズ | ~600MB | ~1GB |

### 将来の拡張オプション

データ量や利用者が増えた場合の対応策：

| 課題 | 対応策 |
|------|--------|
| インデックスサイズ増大 | FAISS IVF（Inverted File Index）への移行 |
| 同時接続数増加 | Gunicorn workerの増加、ロードバランサー導入 |
| 検索速度低下 | GPU版FAISSの導入（faiss-gpu） |

## 将来の拡張構想

### LLM統合（検討中）

```
現状:
  Query → Hybrid Search → Re-rank → 結果リスト表示

将来構想:
  Query → Hybrid Search → Re-rank → LLM → 自然言語で回答
```

社内のLLM APIを想定していますが、コストメリットを考えると現状の検索特化構成がベストプラクティスだと考えています。

### 会社方針との関係

このプロジェクトは個人の現場改善活動としてスタートしましたが、後に会社方針で**Databricks統一**が決定されました。

| フェーズ | アーキテクチャ |
|----------|---------------|
| 個人開発（本リポジトリ） | FAISS + ローカル完結 |
| 会社正式プロジェクト | Databricks + Delta Lake |

本リポジトリのアーキテクチャは、**会社承認前の制約下で極力最適な設計**を頑張った結果です。
Databricks移行後も、ハイブリッド検索やRe-rankingの知見は活用されています。
